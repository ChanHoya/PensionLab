# 연금 인출 전략 엔진 — 상세 개발 계획서 (implementation_plan.md)

> **프로젝트**: 연금컨설팅 앱 — 인출 전략(Decumulation Strategy) 모듈
> **버전**: v1.0 (Draft) | **작성일**: 2026-06-11
> **목적**: 통합연금포털 데이터 기반으로 국민연금·퇴직연금·개인연금의 현행 법규/세제를 반영한
> 개인 맞춤형 인출 전략(순위·기간·금액)을 자동 수립하는 엔진 및 UI 개발

---

## 0. 핵심 설계 원칙

| 원칙 | 내용 |
|---|---|
| **P1. 룰의 외부화** | 세법·건보 기준은 매년 개정됨. 모든 세율/한도/공식은 코드에 하드코딩하지 않고 **버전 관리되는 룰셋(JSON)** 으로 분리 |
| **P2. 재원(Source) 중심 모델링** | 과세는 '계좌'가 아니라 '재원'(비과세 원금 / 이연퇴직소득 / 세액공제분+운용수익) 단위로 결정됨. 데이터 모델의 1급 객체는 재원 |
| **P3. 불확실성의 명시화** | 통합연금포털 데이터만으로 판별 불가한 항목(예: 세제적격 여부, 세액공제 받은 납입 비율)은 **사용자 확인 플로우**로 해소하고, 미확인 상태에서는 시나리오 분기로 결과 제시 |
| **P4. 시뮬레이션 ≠ 자문** | 결과는 "예시 계산"이며 세무 자문이 아님을 모든 출력에 고지 (컴플라이언스 §9) |
| **P5. 가변성** | 어떤 적립액 조합이 입력되어도 동일한 의사결정 프레임워크(§4)가 작동해야 함 |

## 1. 시스템 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                        Frontend (Next.js)                    │
│  업로드/입력 위저드 → 재원 확인 플로우 → 전략 대시보드/비교   │
└──────────────────────────┬──────────────────────────────────┘
                           │ REST/JSON
┌──────────────────────────┴──────────────────────────────────┐
│                      Strategy API Server                     │
│  ┌────────────┐ ┌────────────┐ ┌─────────────┐ ┌──────────┐ │
│  │ M1 Ingest  │→│ M2 Classify│→│ M3 TaxRule  │←│ RuleSet  │ │
│  │ (파서)     │ │ (재원분류) │ │ Engine      │ │ Store    │ │
│  └────────────┘ └────────────┘ └──────┬──────┘ └──────────┘ │
│                                       │                      │
│  ┌────────────┐ ┌────────────┐ ┌──────┴──────┐              │
│  │ M6 Report  │←│ M5 Compare │←│ M4 Simulator│              │
│  │ (리포트)   │ │ (시나리오) │ │ (연도별)    │              │
│  └────────────┘ └────────────┘ └─────────────┘              │
└─────────────────────────────────────────────────────────────┘
```

- **M1 Ingest**: 통합연금포털 엑셀 파서 + 수기 입력 폼
- **M2 Classify**: 계좌 → 재원 분해, 모호 항목 사용자 확인
- **M3 TaxRule Engine**: 룰셋 기반 세액·한도 계산 (순수 함수)
- **M4 Simulator**: 연도별 세후 현금흐름 시뮬레이션
- **M5 Compare**: 전략 시나리오 생성·비교·추천
- **M6 Report**: 그래프/표/내보내기(xlsx, pdf)

기술 스택(권장): Next.js + TypeScript / 계산 엔진은 프론트·백 공유 가능한 순수 TS 패키지(`@pension/engine`)로 분리. 모든 금액은 **정수(원)** 로 처리(부동소수점 금지), 내부 단위 일관성 검사 필수(통합연금포털은 천원 단위 → 적재 시 원 단위 변환).

## 2. 데이터 모델

### 2.1 핵심 엔티티 (TypeScript)

```typescript
/** 연금 재원의 세제 유형 — 과세의 1급 분류 */
type SourceTaxType =
  | 'PUBLIC_PENSION'        // 국민연금 등 공적연금 (무조건 종합과세, 사적과 비합산)
  | 'DEFERRED_RETIREMENT'   // 이연퇴직소득 (퇴직금 재원: 1,500만 한도 무관, 감면율 적용)
  | 'TAX_CREDITED'          // 세액공제 받은 납입분 + 운용수익 (1,500만 한도 대상)
  | 'NON_CREDITED'          // 세액공제 받지 않은 납입원금 (비과세, 한도 무관)
  | 'OLD_PERSONAL'          // (구)개인연금저축 1994.6~2000.12 (비과세)
  | 'NON_QUALIFIED';        // 세제비적격 연금보험 (요건 충족 시 비과세)

interface PensionAccount {
  id: string;
  owner: 'SELF' | 'SPOUSE';
  provider: string;                 // 금융기관명
  productName: string;
  category: 'NATIONAL' | 'RETIREMENT' | 'PERSONAL';  // 통합연금포털 분류
  accountOpenDate?: ISODate;        // 연금수령연차 기산에 필요 (2013.3.1 전후 분기)
  payoutStart: { year: number; month?: number };
  payoutEndAge?: number;            // 종신이면 undefined
  scheduledPayments: YearlyAmount[]; // 포털 예시금액 (연령별)
  sources: SourceBalance[];         // ★ 재원 분해 (합 = 계좌 평가액)
  flexibility: 'FLEXIBLE' | 'CONTRACT_FIXED' | 'UNKNOWN'; // 수령기간 변경 가능 여부
  classificationStatus: 'CONFIRMED' | 'INFERRED' | 'NEEDS_USER_INPUT';
}

interface SourceBalance {
  taxType: SourceTaxType;
  amount: number;                   // 원
  retirementTaxIfLumpSum?: number;  // DEFERRED_RETIREMENT일 때: 일시금 수령 가정 퇴직소득세
}

interface UserProfile {
  birth: { year: number; month: number };
  targetEndAge: number;             // 희망 인출 종료(계획) 나이, 기본 90~100
  desiredAnnualExpense: number[];   // 연령별 목표 생활비 (또는 상수+물가율)
  otherTaxableIncome: YearlyAmount[]; // 임대·근로 등 (종합과세 비교에 필수)
  npsOption: { startAge: number; deferYears: 0|1|2|3|4|5; earlyYears: 0|1|2|3|4|5 };
  healthInsuranceContext: 'DEPENDENT_TARGET' | 'REGIONAL' | 'EMPLOYED'; // 피부양자 희망 여부
}
```

## 3. 모듈별 상세 설계

### M1. 데이터 수집 (Ingest)

**3.1.1 통합연금포털 엑셀 파서**
- 입력: `예시연금액` 시트 (검증된 실제 포맷 기준)
  - 헤더: `단위 : 천원`, 생년월, 연령 행(58세~90세), 행별 `[구분코드, 본인/배우자, 기관명, 상품명, 개시시기, 연령별 금액..., 소계]`
  - 구분코드 매핑: `3=국민연금, 2=퇴직연금, 1=개인연금` (★ 샘플 1건으로 확정 불가 — 추가 샘플 3건 이상으로 검증 후 확정, 미매칭 시 사용자 확인 폴백)
- 처리: 천원→원 변환, 합계 행과 행별 합산 **교차 검산** (불일치 시 경고), 개시 시기 문자열 파싱(`2032년 11월`, `2027년` 혼재)
- 예외: 시트명/레이아웃 변경 감지 → 수기 매핑 UI로 폴백
- 산출: `PensionAccount[]` (classificationStatus = INFERRED)

**3.1.2 수기 입력 폼**: 포털 미사용자·배우자 데이터용. 계좌 단위 입력 + 재원 분해 입력.

### M2. 재원 분류 (Classify) — 본 로직의 분기점

자동 추론 규칙 (확신도와 함께 기록):

| 입력 단서 | 추론 | 확신도 |
|---|---|---|
| category=NATIONAL | PUBLIC_PENSION | 높음 |
| category=RETIREMENT | DEFERRED_RETIREMENT (단, IRP 내 추가납입분 존재 가능) | 중간 → 사용자 확인 |
| 상품명에 "연금저축펀드/계좌" + 2001 이후 | TAX_CREDITED + NON_CREDITED 혼합 | 중간 |
| 상품명 모호 (예: "평생연금저축") | **판별 불가** | NEEDS_USER_INPUT |

**사용자 확인 플로우(필수 UX)** — 모호 계좌마다 3문항 위저드:
1. 이 상품의 가입 시기는? (1994~2000 / 2001~2013.2 / 2013.3 이후 / 모름)
2. 납입 중 연말정산 세액(소득)공제를 받았습니까? (전액/일부/안 받음/모름)
3. 일부라면 공제 받은 누적 납입액은? (금융기관 '연금보험료 등 소득·세액공제 확인서'로 조회 안내)

→ "모름" 응답 시: **시나리오 분기 모드** 활성화 (해당 계좌를 비과세/적격 양 가정으로 병렬 시뮬레이션, 결과 화면에 A/B 동시 표기). 단정 출력 금지(P3).

### M3. 세제 룰 엔진 (순수 함수 라이브러리)

핵심 함수 시그니처 및 로직:

```typescript
// (1) 연금수령한도: limit = balance / (11 - yearIndex) * 1.2, yearIndex > 10이면 무제한
calcWithdrawalLimit(balance, payoutYearIndex, rules): number

// (2) 사적연금 1,500만 한도 판정 — TAX_CREDITED 재원 인출분만 합산
//     (이연퇴직소득, 비과세 재원, 공적연금은 제외)
isOverSeparateThreshold(taxCreditedDrawTotal, rules): boolean

// (3) 적격분 세액: 한도 이내 → 연령별 3.3~5.5% 원천징수로 종결
//     한도 초과 → max 절세 = min(종합과세 시뮬, 16.5% 분리과세) 자동 비교
calcTaxOnCredited(draws, age, otherIncome, publicPensionTaxable, rules): TaxResult

// (4) 퇴직금 재원 세액: 일시금 가정 퇴직소득세 × (1 - 감면율(수령연차))
//     수령한도 초과 인출분은 감면 배제(100% 부과)
calcTaxOnDeferredRetirement(draw, lumpSumTaxRate, payoutYearIndex, overLimit, rules): number

// (5) 국민연금: 과세대상연금액(2002년 이후 납입 비율 — 사용자 입력 or 추정치+불확실 플래그)
//     → 연금소득공제 → 종합과세
calcTaxOnPublicPension(annualAmount, taxableRatio, otherIncome, rules): number

// (6) 건보 판정: 공적연금 연액 > 2,000만 → 피부양자 불가
//     지역가입자 보험료 추정 = f(공적연금×50% 반영 + 기타소득, 재산 입력 시)
assessHealthInsurance(profile, yearlyIncome, rules): HIResult

// (7) 계좌 내 인출 순서 강제: NON_CREDITED → DEFERRED → TAX_CREDITED (법정 순서)
resolveDrawComposition(account, drawAmount): SourceDraw[]
```

**검증 요구**: 함수별 골든 테스트 — 국세청 예시·금융기관 공시 계산기 결과와 대조한 테스트 케이스 ≥ 30건. 종합과세 비교(3번)는 연금소득공제 테이블·인적공제 단순화 가정을 명시하고, 정밀 신고용 수치가 아님을 결과에 표기.

### M4. 시뮬레이터 (연도별 엔진)

- 입력: `UserProfile + PensionAccount[] + Strategy(전략 파라미터) + RuleSet`
- 단위: 1년 스텝, 현재 나이 → `targetEndAge`
- 각 연도 처리 순서:
  1. 공적연금 수령액 확정 (개시/연기/조기 옵션 반영, 물가상승률 가정 적용 — 가정값은 UI 노출)
  2. 전략에 따른 사적연금 계좌별 인출액 결정 (§4 프레임워크)
  3. 연금수령한도 검사 → 초과분 자동 재배분 또는 경고
  4. 재원별 인출 분해(M3-7) → 세액 계산(M3-3,4,5) → **종합 vs 16.5% 자동 선택**
  5. 건보 판정/추정 보험료(M3-6)
  6. 잔액 롤포워드: `balance' = (balance - draw) × (1 + 기대수익률)` (계좌별 수익률 가정, 기본값: 안전자산 2.5%/투자형 4.5% — 사용자 조정 가능)
- 출력(연도별): 세전수령액, 재원별 분해, 세액, 건보료 추정, 세후 현금흐름, 목표 생활비 대비 과부족, 계좌 잔액
- 성능 목표: 1개 시나리오 50년 × 10계좌 < 50ms (순수 함수, 메모리 내 계산)

### M5. 전략 생성·비교 (Strategy & Compare)

**5.1 전략 파라미터 공간**
- 국민연금: 개시 나이 (조기 −5 ~ 연기 +5)
- 계좌별: 인출 개시 나이, 수령 기간(또는 종료 나이), 연간 금액 패턴(정액/정률/한도최대/평탄화)
- 적격분 글로벌 제약: "연 1,500만 이내 유지" ON/OFF

**5.2 기본(추천) 전략 생성 휴리스틱 — §4 프레임워크의 코드화**
```
1. 재원 집계: T_credited(적격), T_deferred(퇴직), T_free(비과세), 공적연금 프로파일
2. 판별식: T_credited / 희망수령연수 ≤ 1,500만?
   YES → 적격분을 1,500만/년 이하로 가장 긴 기간에 평탄화 (70세·80세 세율 하락 활용)
   NO  → (a) 기간 연장 시도 → (b) 부부 분산 제안(배우자 데이터 있을 때)
         → (c) 그래도 초과: 연도별 종합 vs 16.5% 자동 비교로 최소세액 경로 채택
3. 퇴직금 재원: 공적연금 공백기(브리지)에 전진 배치하되
   - 수령한도 내 + 총 수령기간 ≥ 11년(40% 감면), 가능하면 ≥ 21년(50% 감면) 유지
4. 비과세 재원: 최후순위 버퍼 (의료비 등 변동지출 대응)
5. 공적연금: 사적연금 두께·생활비 충족도·절벽 여부로 연기/정시/조기 후보 생성
6. 소득절벽 검사: 어떤 연도의 세후현금흐름 < 목표생활비 × 0.85 → 평탄화 재배분 or 연기 제안
```

**5.3 비교 화면용 표준 시나리오 4종 자동 생성**
- S0: 현재 계약대로 (As-Is, 포털 예시 그대로)
- S1: 절세 평탄화 (추천 휴리스틱)
- S2: S1 + 국민연금 5년 연기
- S3: 사용자 커스텀
- 비교 지표: 생애 총 세후수령액, 총 세액+건보료, 최저연도 현금흐름, 소득절벽 유무, 피부양자 가능 기간

**5.4 (Phase 3) 최적화 옵션**: 휴리스틱 대신 탐색(그리드/유전 알고리즘)으로 생애 세후수령액 최대화. 목적함수에 유동성 페널티·할인율 포함. v1 범위 아님.
